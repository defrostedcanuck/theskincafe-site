  <action>
    Assume production domain is `theskincafe.net` per RESEARCH.md Assumption A1 and STATE.md context. Record this assumption explicitly in the artifact and flag it as a pending client confirmation.

    Run the following dig queries against Google Public DNS (`@8.8.8.8`) so results are independent of any local resolver cache. Capture raw output for each. Use `+noall +answer +ttlid` for clean parsing, and capture the timestamp of capture.

    Commands to run and capture in the artifact, in this exact order:
      1. `dig NS theskincafe.net @8.8.8.8 +noall +answer` — identifies current DNS provider
      2. `dig SOA theskincafe.net @8.8.8.8 +noall +answer` — authoritative zone info + TTL
      3. `dig A theskincafe.net @8.8.8.8 +noall +answer` — apex A record (current host)
      4. `dig AAAA theskincafe.net @8.8.8.8 +noall +answer` — apex IPv6 (should be empty; Vercel doesn't support IPv6)
      5. `dig MX theskincafe.net @8.8.8.8 +noall +answer` — business email records
      6. `dig TXT theskincafe.net @8.8.8.8 +noall +answer` — SPF, Google/M365 verification, DMARC on apex (some orgs put DMARC on apex; canonical is `_dmarc` subdomain)
      7. `dig TXT _dmarc.theskincafe.net @8.8.8.8 +noall +answer` — DMARC per RFC 7489
      8. `dig CAA theskincafe.net @8.8.8.8 +noall +answer` — CA authorization (Pitfall 5: existing CAA without letsencrypt.org blocks Vercel SSL)
      9. `dig CNAME www.theskincafe.net @8.8.8.8 +noall +answer` — www subdomain
     10. `dig A www.theskincafe.net @8.8.8.8 +noall +answer` — in case www is A not CNAME
     11. `dig TXT send.theskincafe.net @8.8.8.8 +noall +answer` — pre-check for any existing Resend/AWS SES records (should be empty)
     12. `dig TXT resend._domainkey.theskincafe.net @8.8.8.8 +noall +answer` — pre-check for existing DKIM
     13. `dig ANY theskincafe.net @8.8.8.8 +noall +answer` — best-effort catch-all (some providers respond; many don't; document what comes back regardless)

    Also capture: `whois theskincafe.net | head -60` to identify registrar (needed for Phase 4 NS flip planning) and domain expiry date.

    Structure the artifact with one fenced code block per command, each preceded by a heading naming the query and followed by a one-sentence interpretation ("Current DNS provider is X", "No existing CAA records — Vercel will auto-add letsencrypt.org in Phase 4", etc.).

    Flag in the artifact's "Unknowns" section:
      - Does the client know their registrar login? (Phase 4 NS flip blocker)
      - Is `theskincafe.net` the correct domain or is a `.com` also in play?
      - Any subdomains the client uses for booking, email forwarding, marketing that didn't show in dig (e.g., GoDaddy email forwarding is often on an A record for the apex + MX)?

    If any dig command errors (NXDOMAIN, SERVFAIL), record the error verbatim — an NXDOMAIN on the apex A record would mean the domain is registered but not pointed anywhere, changing Phase 4 scope.

    Do NOT commit production secrets in this file — dig output and whois are public data and safe to commit.
