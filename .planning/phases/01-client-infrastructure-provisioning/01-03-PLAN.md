    Execute on the dashboard at `https://dash.cloudflare.com/` while logged in to the client's Cloudflare account.

    Step 1 — Add site:
    1. Top nav → "Add a Site" (or dash → "+ Add a site").
    2. Enter the domain exactly as in DNS-BASELINE.md (assumed `theskincafe.net` — use whatever is confirmed in HANDOFF-CUTOVER-CHECKLIST.md open question #7).
    3. Select "Free" plan (Resend + Vercel integration needs DNS only; free plan covers this per RESEARCH.md Standard Stack table).
    4. Cloudflare begins its automatic DNS scan of the existing zone. Wait for scan to complete.
    5. Cloudflare shows its imported records list. **Do NOT click "Continue" yet.**

    Step 2 — Diff imported records against DNS-BASELINE.md:
    Open `.planning/phases/01-client-infrastructure-provisioning/artifacts/DNS-BASELINE.md` side-by-side with the Cloudflare import review UI.

    For each record in the baseline (NS, A, AAAA, MX, TXT, CAA, CNAME):
    - Present in Cloudflare's import? → mark as imported in CLOUDFLARE-ZONE.md.
    - Missing from Cloudflare's import but present in baseline? → manually add via "+ Add record" button in Cloudflare UI. Record this in CLOUDFLARE-ZONE.md under "Manually Added (Not Auto-Imported)."

    Types Cloudflare commonly misses per RESEARCH.md Pitfall 1:
    - Google Workspace TXT verification (`google-site-verification=...`)
    - Microsoft 365 TXT verification (`MS=...`)
    - Custom CNAMEs for booking widgets, vanity redirects
    - DMARC records on `_dmarc` subdomain
    - Email forwarder records

    Do NOT enable orange-cloud proxy on:
    - Apex A records that will later point to Vercel (Vercel uses HTTP-01 cert provisioning; proxying breaks it)
    - Any MX record (proxying breaks mail)
    - TXT records (proxying is not applicable to TXT)
    - The future `resend._domainkey` TXT record (mandatory DNS-only per RESEARCH.md — added in Plan 04)

    Step 3 — CAA audit:
    From DNS-BASELINE.md, identify whether any CAA records exist on the apex.
    - If no CAA records exist: do NOT add one. Vercel auto-adds `0 issue "letsencrypt.org"` on SSL provisioning in Phase 4. Document: "No CAA records. Phase 4 Vercel SSL will auto-add letsencrypt.org."
    - If CAA records exist AND include `letsencrypt.org`: Good. Document current set.
    - If CAA records exist and do NOT include `letsencrypt.org`: ADD a record `Type: CAA, Name: @, Flag: 0, Tag: issue, Value: letsencrypt.org, TTL: Auto`. Document and flag.

    Step 4 — Capture Cloudflare-assigned nameservers:
    After "Continue" past the import review, Cloudflare shows two nameserver values (e.g., `alice.ns.cloudflare.com`, `bob.ns.cloudflare.com`). These are account-specific. Capture both in CLOUDFLARE-ZONE.md under "Cloudflare-Assigned Nameservers (for Phase 4 NS flip)."

    **DO NOT change nameservers at the registrar.** Phase 1 ends here. Phase 4 flips.

    Step 5 — TTL reduction at the current DNS provider:
    This is the most under-appreciated pre-cutover step per RESEARCH.md Pitfall 6.

    Look up current DNS provider from the NS records in DNS-BASELINE.md (Task 1 of Plan 01 identified this via dig + whois). Log into that provider's dashboard (GoDaddy, Namecheap, Squarespace, whoever). Access pathway likely requires client to grant login or screen-share — if Craig does not have the provider login, this step becomes client-blocking and must be surfaced in HANDOFF-CUTOVER-CHECKLIST.md as an additional gate.

    Lower TTL to 60s on all live records at the current provider:
    - Apex A
    - `www` CNAME / A
    - All MX records
    - Any SPF TXT
    - Any existing DKIM / DMARC
    - Any subdomain records identified in baseline

    Do NOT lower TTL on NS records themselves — those are controlled at the registrar, not the DNS provider, and won't matter for the flip.

    Record timestamp of TTL change in CLOUDFLARE-ZONE.md. Phase 4 will wait 24h after this timestamp before performing the NS flip.

    If Craig cannot access the current DNS provider:
    - STOP, do NOT proceed further.
    - Add a new row to HANDOFF-CUTOVER-CHECKLIST.md Client-Blocking section: "Client provides access to current DNS provider (registrar X) so TTLs can be lowered pre-flip."
    - Document the stop in CLOUDFLARE-ZONE.md.

    Step 6 — Final diff pass:
    Run `dig` queries from Plan 01 Task 1 AGAIN, but this time with `@<cloudflare-ns-1>` as the resolver to query Cloudflare directly (even though NS at registrar hasn't flipped, we can query Cloudflare's assigned NS directly to verify the zone):
    ```bash
    dig NS theskincafe.net @alice.ns.cloudflare.com    # replace with actual CF NS
    dig A theskincafe.net @alice.ns.cloudflare.com
    dig MX theskincafe.net @alice.ns.cloudflare.com
    dig TXT theskincafe.net @alice.ns.cloudflare.com
    dig CAA theskincafe.net @alice.ns.cloudflare.com
    dig CNAME www.theskincafe.net @alice.ns.cloudflare.com
    ```
