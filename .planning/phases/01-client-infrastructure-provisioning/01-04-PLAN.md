    Step 2 — Add records in Cloudflare — observing Pitfall 3:
    For each record, in dash.cloudflare.com → client account → theskincafe.net zone → DNS → Records → "+ Add record":

    **CRITICAL:** Cloudflare auto-appends the zone suffix to the Name field. When Resend says Name = `send`, paste EXACTLY `send` (not `send.theskincafe.net`). If pasted as FQDN, Cloudflare stores it as `send.theskincafe.net.theskincafe.net` and verification never completes.

    Record entries (exact):
    1. Type: `MX`, Name: `send`, Mail server: `feedback-smtp.us-east-1.amazonses.com`, Priority: `10`, TTL: Auto, Proxy: N/A (MX not proxyable).
    2. Type: `TXT`, Name: `send`, Content: `v=spf1 include:amazonses.com ~all`, TTL: Auto.
    3. Type: `TXT`, Name: `resend._domainkey`, Content: `<DKIM value from Resend dashboard>`, TTL: Auto, Proxy: **DNS only (grey cloud — mandatory; orange cloud breaks DKIM)**.
    4. Type: `TXT`, Name: `_dmarc`, Content: `v=DMARC1; p=none; rua=mailto:dmarc@theskincafe.net;`, TTL: Auto.

    Step 3 — Verify records with dig BEFORE telling Resend to verify:
    ```bash
    # Query Cloudflare directly (NS at registrar still points at old provider — that's fine, Cloudflare's NS is authoritative for its zone)
    dig MX send.theskincafe.net @<cloudflare-ns-1>
    dig TXT send.theskincafe.net @<cloudflare-ns-1>
    dig TXT resend._domainkey.theskincafe.net @<cloudflare-ns-1>
    dig TXT _dmarc.theskincafe.net @<cloudflare-ns-1>
    ```
    (replace `<cloudflare-ns-1>` with the actual assigned NS from CLOUDFLARE-ZONE.md)

    All four queries must return the expected records. If any is empty or wrong:
    - Empty TXT on `send` → probably pasted as `send.theskincafe.net` → fix by editing record Name to just `send`.
    - Truncated DKIM → Cloudflare truncated the long key; edit and re-paste full value.
    - `_dmarc` appears as `_dmarc.theskincafe.net.theskincafe.net` → same FQDN-paste bug, fix.

    Capture all four dig outputs in RESEND-DOMAIN.md under "Pre-Verification DNS Proof."

    Back in Resend → Domains → theskincafe.net → "Verify." Resend queries from its side.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Resend dashboard → Vercel env store | API key value crosses browser → Vercel CLI/UI → Vercel server encrypted storage |
| Vercel env store → preview deploys | Production-grade RESEND_API_KEY accessible to every Preview deployment (any PR branch) |
| Craig's laptop → `.env.local` | Dev key lives on disk; must never be committed or rsynced |
| Cloudflare zone → internet | DNS records (SPF, DKIM, DMARC) are public by definition |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | Resend API key committed to repo (e.g., accidentally pasted into RESEND-DOMAIN.md) | mitigate | Task 3 acceptance_criteria greps for `re_` token patterns across all committed files and fails if any match. Task action explicitly instructs "never paste into committed file." |
| T-04-02 | Information Disclosure | `.env.local` accidentally committed | mitigate | Task 3 verifies `.gitignore` excludes `.env*` AND verifies `.env.local` not in `git ls-files`. |
| T-04-03 | Elevation of Privilege | Production Resend key lets preview deploys send unlimited email if misused | mitigate | Key is restricted to `theskincafe.net` sending-only permission at Resend dashboard (least privilege). No "full access" permission granted. Free tier's 3k/mo limit caps blast radius. |
| T-04-04 | Spoofing | Missing DKIM or SPF → spoofable email from `theskincafe.net` | mitigate | Task 2 requires all three (SPF, DKIM, DMARC) verified green before plan completes. DMARC p=none now; tighten to p=quarantine or p=reject in Phase 4 after delivery data settles. |
| T-04-05 | Information Disclosure | Rsync clobber overwrites `.env.local` on deploy (global CLAUDE.md memory) | mitigate | `docs/env/resend.md` notes Vercel is source of truth; `.env.local` is scratch only — rsync clobber cannot harm production since production reads from Vercel, not `.env.local`. |
