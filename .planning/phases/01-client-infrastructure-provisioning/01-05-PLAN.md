  <decision>
    How are the Square Developer application and production access token created?
  </decision>
  <context>
    Square Developer applications are tied to a specific Square merchant account. The app (and its production access token) can only be created from within a session logged into the client's Square account. Two paths:

    1. Client self-serves: client logs into squareup.com → Developer Dashboard → creates application → generates production access token with CUSTOMERS_WRITE + CUSTOMERS_READ scopes → shares token with Craig via a secure channel (1Password share, Signal, encrypted email).

    2. Craig co-drives via screen share: client joins a 30-minute video call, shares screen, clicks through the Square Developer UI under Craig's guidance, generates tokens, and Craig seeds them into Vercel in real time without the token ever leaving the screen share.

    Option 2 is faster and more reliable (no client misreads scope name, no "did you click production or sandbox?"). Option 1 is more autonomous for the client but higher-risk for misconfiguration.

    Either way, sandbox token is generated from the same Developer Dashboard but does NOT require the client's login for the sandbox environment — Square's sandbox is provisioned per developer account, not per merchant. However, we'll generate both in the same session for bookkeeping.
  </context>
  <options>
    <option id="option-a">
      <name>Client self-serves, shares token via secure channel</name>
      <pros>No scheduling friction; client owns the action end-to-end; no Craig time.</pros>
      <cons>Higher risk of wrong-scope / wrong-env mistakes; token transport requires secure channel + immediate rotation of any leaked-via-email token.</cons>
    </option>
    <option id="option-b">
      <name>Craig co-drives via screen share</name>
      <pros>Scope and env are guaranteed correct; Craig seeds Vercel in real time so token never persists outside Vercel; token value never leaves the screen share.</pros>
      <cons>Requires scheduling a 30-min call with the client.</cons>
    </option>
  </options>
  <resume-signal>Select: option-a or option-b. The downstream task actions are identical — only sourcing differs. If option-a, also confirm the secure channel (e.g., "1Password share link").</resume-signal>
