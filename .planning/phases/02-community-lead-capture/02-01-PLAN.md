<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| module load → server bundle | Server-only SDK imports (square, resend) must stay server-side; any import from a `'use client'` file would bundle secrets |
| env → module | Process env reads (`SQUARE_ACCESS_TOKEN`, `RESEND_API_KEY`, `SQUARE_API_BASE`) happen at module init; tests must stub, not read real secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | I (Info Disclosure) | src/lib/square.ts, src/lib/resend.ts | mitigate | Singletons live in `src/lib/` (server-only). Server actions in `src/app/actions/*.ts` marked `'use server'` at file top. Client components in Plans 02/03 import only the action functions, NEVER the SDK singletons. Verified by frontmatter `files_modified` separation — no client component touches `src/lib/square.ts` or `src/lib/resend.ts`. |
| T-02-02 | T (Tampering) | SquareClient environment selection | mitigate | RESEARCH.md Pitfall 3: `NODE_ENV` is `production` in Vercel Preview, which would mis-route Preview traffic to real Square. Mitigation: drive `environment` from `SQUARE_API_BASE` exact-match (Phase 1 seeds different URLs per Vercel env). Tests in `src/lib/__tests__/square.test.ts` lock this invariant. |
| T-02-03 | I | Error logs during `new SquareClient({...})` | accept | If token is blank in a test env we ignore it (`?? ''`); in prod we rely on Phase 1 seeding the real value. No PII or secret is ever written to stdout at module init. |
| T-02-04 | I | Vitest config | accept | `vitest.config.ts` has no secrets; jsdom is the only env. `src/test/setup.ts` imports `jest-dom` matchers only. |
</threat_model>
