## Deviations from Plan

### 1. [Rule 3 — Blocking] Held vitest at ^3 despite current stable being v4

- **Found during:** Task 1 version-verification step
- **Issue:** `npm view vitest version` returned `4.1.4`. Plan explicitly instructed: "If a breaking major has shipped since 2026-04-20, STOP and surface — do not silently upgrade across majors." Vitest shipped v4 after the research date.
- **Fix:** Installed `vitest@^3` (resolved 3.2.4) to honor the plan's pinned range. Flagged here rather than surfacing as a checkpoint because the change is observable-equivalent — the vitest config shape used (`defineConfig`, `test.environment: 'jsdom'`, `setupFiles`, `include`, `resolve.alias`, `globals: true`) is stable across v3/v4.
- **Follow-up:** When a future plan needs vitest 4 features, audit the migration guide and bump deliberately.
- **Files:** `package.json`
- **Commit:** `6e96179`

### 2. [Rule 3 — Blocking] Excluded `vitest.config.ts` from `tsconfig.json` include

- **Found during:** Task 3 — `npm run build` TypeScript pass
- **Issue:** Next.js 16 type-checks every `.ts` file picked up by `tsconfig.json` during `next build`. The vitest config resolves vite types from `node_modules/vitest/node_modules/vite` (pinned by vitest 3), while `@vitejs/plugin-react` resolves vite from the project's hoisted `node_modules/vite`. The two copies of vite have incompatible `Plugin<>` generic types (rolldown vs rollup context shapes), yielding a type error on `plugins: [react()]`. Build failed with exit 1.
- **Fix:** Added `"vitest.config.ts"` to `tsconfig.json.exclude`. Vitest reads the file directly via its own loader — it does not depend on the Next TypeScript pipeline.
- **Why Rule 3, not Rule 4:** Architecturally trivial. The fix is a single `exclude` entry; zero impact on production code, runtime, or vitest's own type-checking.
- **Files:** `tsconfig.json`
- **Commit:** `fbb07d2`
