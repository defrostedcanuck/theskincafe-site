<objective>
Establish the Next.js 16 SEO file-convention baseline: programmatic sitemap, robots.txt, and a corrected root `<Metadata>` export with `metadataBase`, title template, Open Graph, Twitter card, canonical, and robots fields. Remove the deprecated `keywords` field. This plan unblocks Plan 02 (OG image + JSON-LD) which requires `metadataBase` to be set before relative OG paths resolve.

Purpose: Without this, every relative OG URL fails the Next 16 build, no crawler discovery file exists, and pages lack canonical/robots hints. This is the foundation SEO-01/02/03 all sit on.

Output: A Next 16-compliant root metadata surface, `/sitemap.xml` serving the homepage, `/robots.txt` pointing crawlers at the sitemap, and a shared `src/app/lib/seo.ts` that downstream plans import.
</objective>

```ts
import type { MetadataRoute } from 'next';
import { SITE_URL } from './lib/seo';

export default function robots(): MetadataRoute.Robots {
  return {
    rules: { userAgent: '*', allow: '/' },
    sitemap: `${SITE_URL}/sitemap.xml`,
    host: SITE_URL,
  };
}
```

Notes per RESEARCH.md:
- Homepage only. Do NOT include hash fragments (`/#services`, `/#locations`) — hashes are not separate URLs for crawlers (Open Question #5).
- No `generateSitemaps` split — this site has < 50,000 URLs. Single default export is correct for Next 16.
- `priority: 1` + `changeFrequency: 'weekly'` — per Pitfall 6, Google ignores priority but inconsistent values hurt trust; `1`/`weekly` for the lone homepage is the defensible choice.
- Do NOT add `disallow` rules for now — the site has nothing to hide from crawlers at this phase.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| crawler → sitemap/robots | Crawlers read `/sitemap.xml` and `/robots.txt`; no user input flows in, no PII flows out. |
| build → metadataBase | `process.env.NEXT_PUBLIC_SITE_URL` is read at build; a hostile env var would produce bad OG URLs, not code execution. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `NEXT_PUBLIC_SITE_URL` env var | accept | Value is controlled by Vercel env-var admin (client-owned per Phase 1). Misvalue degrades SEO but does not execute code; `new URL(...)` will throw on malformed input, failing build loudly. |
| T-03-02 | Information Disclosure | `robots.txt` | accept | No private routes exist yet. Plan will not emit `Disallow` entries that leak private path names. Revisit in Phase 4 if admin routes are added. |
| T-03-03 | Denial of Service | `/sitemap.xml` + `/robots.txt` | accept | Next 16 serves both as statically optimized files; no dynamic DB call, no expensive compute. |
</threat_model>
