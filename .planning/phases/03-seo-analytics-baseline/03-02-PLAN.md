Step A — Create `src/app/components/LocalBusinessJsonLd.tsx`. This is a SERVER component; do NOT add `'use client'`.

```tsx
import { SITE_URL } from '../lib/seo';
import { LOCATIONS, type Location } from '../lib/locations';

function toEntity(loc: Location) {
  return {
    '@type': 'DaySpa',
    '@id': `${SITE_URL}/#${loc.slug}`,
    name: loc.name,
    url: `${SITE_URL}/#${loc.slug}`,
    image: loc.image,
    telephone: loc.telephone,
    priceRange: loc.priceRange,
    address: {
      '@type': 'PostalAddress',
      streetAddress: loc.streetAddress,
      addressLocality: loc.addressLocality,
      addressRegion: loc.addressRegion,
      postalCode: loc.postalCode,
      addressCountry: 'US',
    },
    geo: {
      '@type': 'GeoCoordinates',
      latitude: loc.geo.latitude,
      longitude: loc.geo.longitude,
    },
    openingHoursSpecification: loc.hours.map((h) => ({
      '@type': 'OpeningHoursSpecification',
      dayOfWeek: h.days.length === 1 ? h.days[0] : h.days,
      opens: h.opens,
      closes: h.closes,
    })),
    sameAs: loc.sameAs,
  };
}

export function LocalBusinessJsonLd() {
  const jsonLd = {
    '@context': 'https://schema.org',
    '@graph': LOCATIONS.map(toEntity),
  };

  // Per node_modules/next/dist/docs/01-app/02-guides/json-ld.md: escape "<" to block XSS
  // if any property ever becomes user-influenced (future reviews/testimonials).
  const html = JSON.stringify(jsonLd).replace(/</g, '\\u003c');

  return (
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: html }}
    />
  );
}
```

Hard rules:
- NO `'use client'` directive — rendering `<script>` via `dangerouslySetInnerHTML` is the official Next 16 pattern and must stay server-side.
- `.replace(/</g, '\\u003c')` scrub is mandatory (RESEARCH.md anti-patterns section).
- Use `@graph` with stable `@id` per entity (Pitfall 2) — NEVER emit two loose sibling JSON-LD blocks; NEVER use a single `LocalBusiness` with multiple `address` entries.
- Do NOT install `schema-dts` for this phase — RESEARCH.md lists it as optional; input types already enforce correctness.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JSON-LD serialization → HTML | `JSON.stringify` output is injected into the DOM via `dangerouslySetInnerHTML`. If any property ever becomes user-supplied, `<` embedded in that value could close the `<script>` tag and execute arbitrary JS. |
| OG image file → social-platform fetch | Social platforms (iMessage, Slack, FB, LI) fetch the PNG bytes over the public internet; nothing travels back to the site. No PII exposure. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering (XSS) | `LocalBusinessJsonLd.tsx` | mitigate | `JSON.stringify(...).replace(/</g, '\\u003c')` per Next 16 official JSON-LD guidance. Escapes `<` into the JSON `\u003c` escape, preventing any property value from closing the host `<script>` tag. Verified in task automation (grep for `u003c`). |
| T-03-05 | Information Disclosure | `geo` coordinates, `telephone` | accept | All data is already public (business phone, street address, published hours). Geo rounds to the building. No private info leaks. |
| T-03-06 | Spoofing (brand impersonation via OG) | `opengraph-image.png` | accept | File is statically compiled into the build and served from the client's Vercel origin. Platforms fetch from the canonical URL; attackers impersonating the site would need to first compromise DNS (out of Phase 3 scope). |
| T-03-07 | Tampering (wrong `sameAs` Instagram URL) | `locations.ts` | mitigate | Instagram URL was verified from `Footer.tsx` line 49 (`theskincafe`, not `theskincafeaz`) to prevent misattributing the business's social presence to a possibly-unrelated account. |
</threat_model>
